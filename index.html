Introduction

Somewhere outdoors, a phone buzzes in a pocket. A neural network is training on a laptop GPU back home — no one sitting in front of it, no one watching the screen. The results push to GitHub automatically. The person walking doesn't need to be there. They trained a neural network from their phone while on a walk, and it just worked. That one sentence is why everything in this project exists.

A blank terminal. A cursor flickering. /remote-control-protocol. Eight conditions check themselves. A laptop in one room talks to a phone on a sidewalk ten minutes away, and a neural network begins training on a GPU that no one is sitting in front of.

The neural network itself was deliberately small — a 3-layer MLP with 4096 hidden units per layer, trained on MNIST with Adam at lr=0.001 for 8 minutes on the RTX 2000 Ada. None of those parameters were chosen for accuracy. They were chosen for feedback speed. The core philosophy driving every design decision was: quick verifiable output drives design, not the other way around. Checkpoint plots pushed to GitHub every 2 minutes — colorblind-safe Okabe-Ito palette, phone-readable at 150 DPI, two panels showing loss and accuracy against elapsed time. Those plots were not decoration. They were the entire feedback system. When you're on a sidewalk and your terminal is invisible, a plot appearing on GitHub is the only proof that training is alive. If a checkpoint took too long to arrive, the design was wrong — not the interval. The final suite pushed 5 plots after training ended: training curves, easy wins, edge cases, confident failures, and a confusion matrix. Every run was scoped by a tag — one tag, one experiment, one set of outputs — so no run could overwrite another. The model reached 98%+ validation accuracy, but the accuracy was never the point. The point was that the results arrived on a phone, on schedule, while the person who launched them was a quarter mile away and walking.

The human's specific, slightly absurd goal of training an MNIST neural network on a laptop's RTX 2000 Ada GPU, remotely, from a phone, while walking 100,000+ steps outside in ~3 days is how this protocol was born. Claude Code's remote control feature made the connection possible. But the connection alone wasn't enough — CUDA died on suspend, drivers didn't load at boot, sessions dropped and killed running jobs, private repos couldn't be checked from a phone. Each failure became a rule. Each rule became a step. The steps became RCP — a bottom-up approach followed to its finest.

The neural network training was never the project. It was the starting point, the experimental testbed, the breeding ground, and the killer demo — all at once. RCP — Remote Control Protocol — is what emerged.

The Abstraction Ladder

To understand why RCP matters, it helps to see where remote control sits in a longer story. Computing has always moved in one direction: removing things the human used to do manually. Each generation of tools took away a constraint and handed it to the machine. That progression — from Assembly to AI agents — forms a ladder. And remote control is the rung at the top.

Read the ladder from the bottom up. Assembly freed you from binary. C freed you from tracking registers. Python freed you from type declarations and compilation. NumPy freed you from writing loops for math. IDEs freed you from managing files. LLMs freed you from writing code yourself. Cursor put the AI inside the editor so you could stop copy-pasting. Claude Code removed the editor entirely — you talked to it in a terminal, and it read your codebase, wrote code, ran commands, spun up agents. Each step up, something you used to do manually became something the machine handled.

And then the obvious question: if all you're doing is talking to Claude Code, why do you need to sit in front of your computer to talk to it? You don't. That's remote control. That's the top of the ladder — the final manual constraint, physical presence, falls away. You're on a sidewalk with a phone. Your laptop is in another room training a neural network. Claude Code is the one sitting at the keyboard.

But reaching the top of the ladder and staying there are different things. The top rung is real — Claude Code's remote control feature makes it possible — but it's unstable without guardrails. CUDA dies on suspend. Drivers silently fail to load at boot. Sessions drop and kill running jobs. Private repos return 404 from a phone. Every one of these failures happened during the walks that produced RCP. The protocol is what turns a fragile top rung into a platform you can actually stand on. RCP is a preflight checklist for remote-controlled compute sessions. It formalizes the handshake between a human with a phone and an AI agent running on their machine: is the laptop plugged in? Is CUDA alive? Will the GPU survive a sleep event? Is the phone connected? Eight booleans, AND'd together. All green means go. The diagram above is the protocol itself — eight conditions radiate from a central laptop, each connected by a line that glows green when the condition is met. Click any node to toggle it. When all eight are green, the banner flips to READY and the laptop pulses. The machine is safe to walk away from.

The conditions aren't arbitrary. They form a dependency chain. Power keeps the machine alive. WiFi enables the remote session and the git pushes that deliver results. Sleep Inhibit prevents the OS from suspending the GPU mid-training — CUDA contexts don't survive suspend, and recovering from a destroyed context requires a full reboot. The GPU Driver step loads nvidia_uvm, a kernel module that doesn't auto-load on this hardware. CUDA verification confirms the entire stack is functional. Git Public ensures the repo is accessible from a phone browser. GWS CLI creates optionality for communication via Google Drive and Gmail — apart from providing an alternative channel, it provides a backup if the primary connection drops. Phone Connected is the final gate — the human confirms they have a working link before leaving the room.

Design Philosophy

We call the core design insight the Friction-Forged Protocol. The idea is that the most reliable operational protocols aren't designed upfront from requirements documents — they crystallize from failure.

Sleep Inhibit exists because CUDA died on suspend during an early test run and the GPU context couldn't recover without a full reboot. The GPU Driver step exists because nvidia_uvm silently fails to load at boot on this specific hardware, and we only discovered this after torch.cuda.is_available() returned False with no useful error message. Git Public exists because a private repo looked fine from the laptop but returned 404 on the phone. Detached processes exist as free technical insurance — they aren't strictly necessary, but they're good practice that costs nothing and prevents job death if a session drops. Each rule traces to a specific incident.

The result is a protocol that's small, specific, and hard to break — not because someone thought carefully about what might go wrong, but because someone lived through what actually did.

RCP and MCP

RCP rhymes with MCP, and not just phonetically.

MCP — Model Context Protocol — is an open standard by Anthropic that gives AI applications a universal way to connect to external tools and data sources: a client-server architecture that replaces fragile, one-off integrations with a single protocol, the way USB-C replaced a drawer full of proprietary cables. RCP applies the same structural idea to a different problem — not "can the AI reach the tool?" but "can the human control their machine remotely, with Claude Code, from their phone?" Both are protocols. Both replace ad-hoc, fragile processes with a formalized standard so that downstream work can assume a known-good state.

The naming wasn't intentional. RCP was written out in long-form — "Remote Control Protocol" — and the phonetic echo only became obvious after building a local MCP server in a separate project. They operate at different scales but solve the same category of problem: formalize a connection pattern so it's repeatable and reliable. MCP is an ecosystem-wide standard shipped as infrastructure. RCP is a personal protocol, portable across projects, built by one person who wanted the same reliability guarantees for kicking off GPU training from a phone on a walk.

A Claude Code Project

RCP borrows from established categories without belonging to any of them. It has the motivation of internal tooling — encode operational knowledge into something repeatable. The structure of SRE runbooks — phased checklists with verification gates. The safety philosophy of aerospace preflight procedures — all conditions must pass before go. But none of these map cleanly, because all of them assume a human operator.

RCP inverts that. Claude drives the sequence. The human provides judgment at critical gates — confirming they've left the machine, providing a tag name, saying "go" to launch the job. The AI is the operator. The human is the approver. That partnership didn't exist before Claude Code, and there's no established category for tooling that supports it.

This makes RCP a Claude Code project for a simple reason: remote control is a feature unique to Claude Code. It's not available in Cursor, Codex, or any other AI coding tool. RCP exists because remote control exists, and remote control only exists in Claude Code. That's the whole explanation.
